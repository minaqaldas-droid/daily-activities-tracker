# Master Web App Build Prompt

عايزك تساعدني نظبط المشروع ده **Web App احترافي وقابل للتوسع والصيانة** في مجال:  
`Date entry لل Daily activities في الشغل لتكوين history اقدر ارجعله أي وقت واعمل analysis لل data`

---

## التقنيات الأساسية المطلوبة

- **Frontend:** React + TypeScript + Vite
- **Backend / Database / Auth:** Supabase
- **Hosting / Deployment / Serverless Functions:** Cloudflare
- **Version Control:** Git  

### متطلبات أساسية
- لازم المشروع يكون **scalable, maintainable, secure, و production-ready** قدر الإمكان
- لازم يكون **Responsive** بشكل ممتاز على الموبايل و الديسكتوب

---

## أولويات الشغل

1. UI/UX يكون **مبهر جدًا** و **premium**
2. **Responsive on mobiles**
3. Architecture نظيفة وسهلة التوسع
4. Maintainability عالية
5. تنظيم الملفات والكود والـ SQL بشكل احترافي
6. توثيق كل خطوة بحيث نقدر نكمل أو نعيد بناء المشروع بسهولة في أي وقت

---

## طريقة شغلك المطلوبة

- قبل التنفيذ، ابدأ بفهم فكرة المشروع وتحليلها بشكل عملي
- جهّزلي خطة مفصلة خطوة بخطوة للتنفيذ من البداية للنهاية
- اقترح تحسينات إضافية ذكية حتى لو أنا لم أطلبها صراحة، طالما مفيدة للمشروع
- وضّحلي في كل مرحلة: هنعمل إيه، ليه، فين، وإزاي
- امشِ بالتنفيذ بشكل تدريجي ومنظم، بدون ما تبوّظ أي حاجة شغالة
- أي تعديل أو إضافة لازم يكون محافظ على استقرار المشروع
- فكّر دائمًا بعقلية منتج حقيقي سيتباع أو سيُستخدم فعليًا

## التوثيق الإجباري داخل المشروع

لازم من أول خطوة تنشئ وتحدّث ملف Markdown داخل المشروع يكون مرجع ثابت لنا، وليكن مثلًا:

- `PROJECT_CONTEXT.md`

### ويحتوي دائمًا على:

- وصف المشروع
- الهدف منه
- التقنيات المستخدمة
- الهيكل العام للمشروع
- القرارات المعمارية المهمة
- الجداول والعلاقات
- الصفحات والموديولات
- أي ملاحظات أو assumptions
- ما تم تنفيذه بالفعل
- الخطوات التالية
- أي dependencies أو إعدادات مهمة
- أي مشاكل تم حلها وكيف اتحلت

### ولو احتجت، أنشئ أيضًا:

- `CHANGELOG.md`
- `README.md`
- `DEPLOYMENT.md`
- `DATABASE.md`

## آلية التنفيذ في كل مرة

- في كل رد، اذكر باختصار ما الذي عدّلته
- أي SQL queries أو migrations لازم تكون منظمة، مرتبة، ومقسمة بوضوح
- سمّي ملفات SQL بشكل احترافي وواضح
- حافظ على ترتيب الملفات وعدم العشوائية

---

## قواعد مهمة جدًا أثناء التنفيذ

- لا تكسر أي جزء شغال
- لا تغيّر حاجة مستقرة إلا لو في سبب واضح
- لو في أكثر من اختيار تقني، اختَر الأنسب من حيث البساطة والجودة وقابلية الصيانة
- اكتب كود نظيف وواضح وسهل التعديل لاحقًا
- راعِ reusable components
- راعِ performance
- راعِ accessibility بشكل مناسب
- راعِ security best practices
- راعِ future scaling
- تجنّب التعقيد غير الضروري
- لو في حاجة محتاجة قرار، اقترح أفضل قرار مع سبب واضح

---

## بالنسبة للتصميم

- أريد تصميم **premium / modern / elegant**
- اهتم جدًا بالـ spacing والـ hierarchy والوضوح
- اختَر ألوان وفونتات مناسبة لطبيعة المشروع
- استخدم animations **subtle** وراقية وليست مبالغ فيها
- التصميم لازم يكون مريح بصريًا وسهل الاستخدام
- لازم كل شاشة تكون محسوبة للموبايل بشكل ممتاز
- اهتم بحالات:
  - loading
  - empty states
  - error states
  - success feedback
- اجعل الـ UX عملي وسريع وواضح

## مهم

- رتّب الـ SQL في ملفات واضحة ومنظمة كل مرة
- حدّث ملفات التوثيق مع كل مرحلة
- تعامل مع المشروع كأنه منتج حقيقي احترافي قابل للبيع والتطوير لاحقًا

---

## تعليمات إضافية مهمة

- حافظ دائمًا على أن أي feature جديدة تكون قابلة لإعادة الاستخدام وقابلة للتطوير
- عند تنفيذ أي جزء، فكّر في edge cases وempty states وerror handling
- أي صلاحيات أو أدوار users يجب التفكير فيها من البداية إذا كانت مفيدة
- اقترح naming conventions واضحة للملفات والجداول والمكونات
- لو في قرار بين حل سريع وحل أنظف طويل المدى، وضّح الفرق واختر الأنسب
- عند تعديل أي شيء، لا تغيّر ملفات أو أجزاء لا تحتاج تعديل
- عند كل مرحلة، اكتب لي **next steps** بوضوح
- لو وجدت تضاربًا في المتطلبات، اختَر الحل الأكثر منطقية واستقرارًا واذكر السبب
